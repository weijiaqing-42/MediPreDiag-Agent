# MediPreDiag-Agent 面试背诵话术（白话版）

> 使用说明：以下所有回答均为口语化表述，可直接背诵。对项目中的不足之处已按最优方案修正过，面试时以此为准。

---

## Q1: 简单介绍一下项目，整体架构？

这个项目叫 MediPreDiag-Agent，是一个面向医疗预诊场景的多智能体协作系统。说白了就是用户用自然语言描述自己的症状，系统通过好几个专门干不同事的 Agent 一起协作，最后给出症状分析、严重程度怎么判断、用药建议、以及附近去哪看病的推荐。

架构是五层的：

第一层是接入层，用的 FastAPI，同时开了 REST 和 WebSocket 两个通道。REST 给那种集成调用的场景——比如别的系统调我查一下症状，一问一答就完了。WebSocket 是给前端用的，因为 Agent 跑起来分好多步，用户等着的时候能看到每一步的状态推送，体验好很多。

第二层是编排层，用 LangGraph 做的。把整个工作流建模成一张有向图，每个节点是一个 Agent 函数，节点之间的边分成两种：普通边就是固定流转，条件边根据运行时的状态动态决定下一跳走哪。整个图是从 memory_loader 开始的，先把用户之前的会话上下文加载出来，然后进意图分类器判断用户想说啥，再根据意图路由到不同的链路。

第三层是 Agent 层，六个专职 Agent：意图分类器、症状提取、严重度评估、医疗建议生成、药品问答、地点推荐。每个 Agent 只干一件事，prompt 很短很精准。除了地点推荐走的是 MCP 协议调高德地图 API 不需要 LLM 之外，其余五个都是调大模型做推理。

第四层是记忆和存储层，用了三个数据库：Redis 做短期会话记忆，TTL 设为 30 分钟；Milvus 向量数据库做长期记忆，把用户的诊断摘要向量化存起来，下次来的时候做语义检索召回相关历史；MySQL 做结构化持久化，存用户档案、诊断记录这些。

第五层是检索层，也就是 RAG 的核心链路。用户的问题先进来做 embedding，然后两路并行：一路走 Milvus 的 HNSW 索引做向量检索，一路走 BM25 做关键词稀疏检索。两边出来的结果用 RRF 算法融合排名，然后做 content MD5 去重，最后交给 DashScope 的 qwen3-rerank 做语义重排序，取 top-5 注入给医疗建议 Agent 当上下文。

部署的话就是 Docker Compose 一把起，底层五个容器：MySQL、Redis、Milvus、etcd、MinIO。

## Q2: 接入层怎么做的？FastAPI 了解吗？REST 和 WebSocket？

FastAPI 我比较熟，它是基于 Starlette 的异步 Python 框架，最大的几个卖点就是自动生成 Swagger 文档、Pydantic 做请求体校验、原生支持 async/await。我用下来感觉比 Flask 适合做异步服务，比 Django 轻很多。

接入层我主要做了三件事：

第一是生命周期管理。FastAPI 有个 lifespan 机制，可以写一个异步上下文管理器，在服务启动的时候按顺序初始化各个组件。我的做法是 Redis 先进——因为短期记忆每次请求都要读写，然后是 Milvus 初始化 Collection，最后是 MySQL 建表。每个初始化我都包了 try/catch，比如 Redis 连接设了 5 秒超时，连不上就打一行 warning，服务照起不误。关闭的时候做对应的资源释放。这就是一个优雅降级的设计——宁可服务缺胳膊少腿跑起来，也不要启动阶段直接崩掉。

第二是 REST 端点。路径是 `/api/v1/chat`，请求体用 Pydantic 的 ChatRequest 做校验，里面有 user_id、message、还有可选的 session_id、image_url、user_location。没有 session 的话自动生成 UUID。然后核心逻辑很简单——把请求体展开成一个 TypedDict 叫 MediState，这个是整个图谱的共享状态对象，然后直接 `graph.ainvoke` 丢给 LangGraph 去跑。跑完之后拿到最终的 state，映射成 ChatResponse 返回给客户端。异常的话就是 HTTP 500。

第三是 WebSocket 端点。WebSocket 本质是 TCP 上的全双工协议，不像 REST 那样请求-响应单向，它是建立连接以后两边随时能互发消息。我用它的原因很直接——Agent 图谱跑起来有好几步，意图分类、症状提取、RAG 检索、严重度评估、医疗建议生成，每一步都有延迟，REST 模式用户得等全部跑完才看到结果。WebSocket 模式下，连上之后先回一条 `status: Processing`，让前端知道已经开始跑了，最终结果用 `type: response` 推送过去。循环里还处理了 WebSocketDisconnect 异常，客户端断开不会崩服务。

两种通道我自己都写了。REST 适合集成调用，WebSocket 适合前端交互，两边共享同一套 MedState 初始化和 graph.ainvoke，核心逻辑没有重复代码。

## Q3: 降级启动怎么设计？组件挂了怎么办？

这个我做得比较完善了，分了启动层和运行时两层来处理。

启动层的话，前面说了 lifespan 里每个组件都包 try/catch。Redis 用 asyncio.wait_for 给了 5 秒超时，连不上就打 warning 继续往下走。Milvus 和 MySQL 也一样。这个设计的原则就是"宁可降级不可崩"——Redis 挂了服务照起，只是没有会话记忆；Milvus 挂了服务照起，只是 RAG 检索不可用。

运行时这层更关键。每个 memory 读写操作我都包了 try/catch，比如说 memory_loader_node 里面加载 Redis 会话历史，异常的话就设成空列表，主流程不受影响继续往下跑。memory_updater_node 写 Redis 和 Milvus 也一样，写不进去就打日志，对话结果照样返回给用户。

RAG 检索这块我也做了 fallback。如果 Milvus 向量检索挂了，会捕获异常设成空结果，然后 BM25 那路单独跑——因为 BM25 索引我维护在全库上，不依赖 Milvus，一个人也能兜底。如果两边都挂了，还有一层兜底——检索异常时设 rag_context 为空字符串，医疗建议 Agent 就会基于纯 LLM 知识来回答，只是会在回答里加一句"当前知识库不可用，建议仅供参考"。这样从启动到运行形成了一个完整的降级链路。

## Q4: 简历里的业务中断/回滚/超时降级机制？

这块我三个都实际落地了，一个 InterruptHandler 类统一管：

**超时降级**——config 里设了 `node_timeout: 15`，每个 LLM 调用的节点都包了 `interrupt_handler.execute_with_timeout`，底层是 `asyncio.wait_for`。比如医疗建议生成节点，超过 15 秒就抛出 TimeoutError，被捕获后设 `timeout_flag = True`，然后图的条件路由看到这个 flag 就跳到 interrupt_handler 节点生成兜底回复。这样任何一个节点卡住都不会无限等。

**重试**——`execute_with_retry` 方法，三个关键节点都挂了：意图分类器（调外部 API 最容易抖）、症状提取（结构化抽取不能错）、RAG 检索（Milvus 查询可能偶发失败）。每个都是 3 次重试 + 递增退避等待，第一次失败等 1 秒、第二次等 2 秒、第三次等 3 秒，三次全失败才标记异常。

**中断**——分两种。被动中断就是关键词匹配，用户说"取消""不用了"这些，在意图分类节点第一步就检查，命中设 `interrupt_flag = True`，所有条件路由优先检查这个 flag。主动中断的话，我做了一个 WebSocket 通道的实时取消机制——前端有个停止按钮，按了之后通过 WebSocket 发 `{"type":"cancel"}`，后端用 `asyncio.Event` + `Task.cancel()` 来实现。具体就是图文执行和取消监听两个协程同时跑，哪个先完成另一个就被取消。如果用户按了取消，`task.cancel()` 会在当前节点的 await 点抛出 CancelledError，每个节点都捕获这个异常、设 interrupt_flag、正常 return，图继续走到 interrupt_handler 给用户一个"对话已取消"的回复。取消的同时把当前状态写进 Redis，下次用户回来能恢复上下文。

**回滚**——每个节点在执行前会把当前 node 名写进 state 的 `rollback_target`字段。如果后续节点检测到 timeout 或 interrupt，路由函数就根据这个字段决定回滚到哪个节点，并且 interrupt_handler 根据 rollback_target 生成对应的兜底回复（比如"症状分析暂时不可用，建议咨询专业医生"和"地点推荐暂时不可用"是不一样的）。Redis 里存了完整的 state 快照，回滚后能恢复到之前的状态重新走。

## Q5: 编排层用 LangGraph？还了解哪些框架？

对，编排层选了 LangGraph。它是 LangChain 团队出的框架，核心思想是把 Agent 工作流建模成有向图。每个节点就是一个函数，可以是调 LLM、调工具、或者纯 Python 逻辑。边分两种：普通边固定往下走，条件边根据运行时的 state 动态决定下一跳。底层是状态图而不是纯 DAG，支持循环，维护一个共享的 TypedDict 做状态在各个节点之间流转。

我的用法就是很标准的 LangGraph 模式：StateGraph 建图 → add_node 挂节点 → add_conditional_edges 做条件路由 → compile 编译。六个条件路由函数都只读 state 里的字段来决策，没有额外 LLM 开销。

其他框架我也了解不少：

CrewAI，角色扮演式的多 Agent，每个 Agent 有 role、goal、backstory，概念很直观上手也快，但不支持复杂的分叉和循环，我这种医疗场景分支太多了不适合。

AutoGen，微软出的，对话驱动的多 Agent，Agent 之间通过消息传递协作，支持 group chat。优点是微软生态好，代码生成场景深耕，但图级别的流程控制不如 LangGraph 灵活。

MetaGPT，用软件工程 SOP 模拟 Agent 协作，把 Agent 映射成 PM、架构师、工程师这些角色，按需求→设计→编码→测试的流水线跑。理念很有意思但是太重了，适合演示级别的复杂项目。

Dify 和 Coze 这些低代码平台也用过，拖拽式编排上手很快，适合快速验证想法，但深度定制能力受限。

手写编排我也做过，就是用 async/await 串行或者 asyncio.gather 并行。如果 Agent 拓扑简单，手写反而是最轻量的。我这个项目分支多、需要持久化和中断恢复，所以选 LangGraph 是最合适的。

## Q6: 为什么用多智能体？每个怎么设计？

两个核心原因。第一个是职责单一——如果把所有逻辑揉进一个大 Agent，prompt 会特别长，既要意图识别又要症状提取又要严重度评估又要药品查询又要地点推荐，每一步都得写一堆规则，LLM 很容易多任务串扰。第二个是可控性——拆开之后每个节点只干一件事，出问题立刻能定位到是哪个 Agent 错了，debug 效率高很多。

设计上我用了统一的模式：每个 Agent 就是一个异步函数，输入是完整的 MediState，输出是修改过的 MediState，因为 LangGraph 的状态是共享的。这样做的好处是上下游之间不需要显式传参数——比如症状提取写完了 `possible_diseases`，下游医疗建议直接读 state 就行，代码耦合很低。

具体来说：

第一个是**意图分类器**，用的是 qwen-turbo 加 Few-shot 提示词，做 6 分类，输出严格 JSON。temperature 设 0.0，max_tokens 卡在 64，保证输出稳定。

第二个是**症状提取 Agent**，用 qwen3-max，从用户描述和 RAG 结果里结构化提取症状列表、疑似疾病列表以及置信度，还有严重度初步 hint。temperature 设 0.3，保留一点弹性但不大。

第三个是**严重度评估 Agent**，我设计了一个双重机制。先用关键词预筛——胸痛、呼吸困难、大出血这些 19 个紧急关键词做精确匹配，命中直接标 severe，不走 LLM。为什么这么设计？因为紧急情况不能赌 LLM 的稳定性。没命中才调 qwen-turbo 做分级。

第四个是**医疗建议 Agent**，这是最核心的节点，用 qwen3-max。prompt 最长，因为我注入了七块上下文：用户描述、提取的症状、疑似疾病、严重度、历史对话、长期健康摘要、RAG 知识库结果。要求输出四个段落：分析、建议、危险信号、免责声明。temperature 0.5 给生成一些灵活性。

第五个是**药品问答 Agent**，用的 Tool Use 模式。先调一次 LLM 做药名提取和问题类型分类（用法、禁忌、副作用等），拿到药名后并发走两路——一路从 Milvus 的知识库里查药品说明书，一路用 RAG 混合检索补充相关药品知识，两个数据源拼在一起给主 LLM 生成回答。最后必须带免责声明。

第六个是**地点推荐 Agent**，这个比较特殊，不走 LLM，而是通过 MCP 协议调用高德地图 POI API。根据严重程度来选推荐类型——mild 只推荐药店、moderate 推荐医院和诊所、severe 和 emergency 推荐急诊和医院，搜索半径也不同。

## Q7: Agent 范式了解吗？

了解，主要的几类我都清楚：

ReAct 是最经典的，Thought-Action-Observation 三个步骤交替循环，推理和行动交错。每一步看到工具返回的结果之后再决定下一步怎么做，好处是灵活，坏处是循环次数不可控。

CoT，纯推理不去调工具，就是让 LLM 把中间过程写出来，一步步思考再给最终答案。适合数学推理这种不需要外部知识的问题。

Plan-and-Execute，先做全局规划再逐步执行。第一步出完整计划，第二步按顺序执行。好处是全局视角，坏处是计划静态，中间错了不好修正。

Function Calling，这是现在生产最主流的。不是让 LLM 自由决定调用啥，而是先定义一个 functions schema 注册给模型，模型输出结构化的函数名加参数 JSON，外部 runtime 执行函数把结果传回去。OpenAI Function Calling、各家 MCP 协议都走这个。

Reflection，生成完答案之后再自我检查和修正，多迭代几轮，适合高可靠性场景。进阶版叫 Reflexion，把反思结果存长期记忆避免重复犯错。

Multi-Agent/Supervisor，就是我这个项目用的范式——多个 Agent 各司其职，由一个 Supervisor 统一调度，每个子 Agent 有独立的角色和 prompt。

Tree of Thoughts，把 CoT 的线性推理扩展成树状多路径探索，对每条路径打分，高分保留低分剪掉。解决需要探索和回溯的复杂问题，但一次要跑多次 LLM 调用，成本高。

我这个项目更接近 Multi-Agent 加条件路由的混合——不走典型的 ReAct 循环，因为医疗预诊的流程天然是流水线式的。但如果做多轮追问那种交互式病情收集，那就更像 ReAct 了。

## Q8 + Q9: RAG 混合检索和重排序，链路为什么这么设计？

整个 RAG 链路我设计成了漏斗式逐步收紧：宽召回 → 融合去重 → 精排 → 注入。每一层都在缩小候选集同时提升精度。

第一步是双路并行检索。为什么两路？因为医疗查询有两种特点——一是术语专有名词多，比如"阿莫西林""房颤""糖化血红蛋白"这些，二是语义有模糊性，比如"头疼"可能是偏头痛也可能是高血压。向量检索擅长语义，能召回"血管性头痛"这种近义词；BM25 擅长精确词匹配，遇到专有名词直接命中。并行是因为两边互不依赖，同时跑不增加任何延迟。我这边 BM25 的索引是维护在全库上的，不是从向量结果里即时构建的——那样会丢召回率。我是在灌库的时候同时构建 BM25 索引并持久化，检索的时候两路真正并行。

第二步是 RRF 融合。为什么选 RRF 不选线性加权？因为两种检索的分数尺度完全不一样——向量返回 0 到 1 的余弦相似度，BM25 是 IDF 累加的正实数，不能直接相加，强行归一化加权也比较玄学。RRF 不管分数绝对值只看排名，公式是 1/(60+rank)，跨系统天然可比，零超参不用调，业内验证也是最充分的。

第三步是 Content MD5 去重。同一个文档可能向量检索和 BM25 都命中了，ID 不同但内容完全一样，RRF 融合后两条都在结果里。所以取 content 的 MD5 哈希，相同哈希只保留 RRF 分数更高的那条。125 篇的规模下碰撞概率基本为零。

第四步是重排序，用的 DashScope 的 qwen3-rerank API。RRF 只给了基于排名的粗排序，A 排第 1 B 排第 5，不代表 A 比 B 语义上紧多少。Rerank 做的是 cross-encoding——把 query 和每个文档拼成一对一起输入 Transformer 做双向交叉注意力，输出的相关性分数才是真正语义层面的。这个操作精确但贵，所以我把它放在漏斗最后只跑 30 条候选，延迟控制在 100 毫秒左右。

第一次做的时候我踩了个坑——用 qwen-turbo 做 LLM 排序，把所有候选文档塞 prompt 里让 LLM 挑。结果延迟高到差不多 1 秒，而且排序结果不稳定。后来想明白了——重排是检索场景，要的是快和一致，不需要生成能力，专用小模型就是干这个的。换成 DashScope 的 Rerank API 之后延迟从 1 秒降到 100 毫秒左右，排序也稳定了。

整体延迟：embedding 大概 50 毫秒，Milvus 大概 30，BM25 大概 10，RRF 不到 1 毫秒，Rerank 大概 100 毫秒，端到端差不多 200 毫秒，在对话场景里用户基本感知不到。

Embedding 选的是 text-embedding-v3，1024 维，支持指定维度。没选开源的 BGE-m3 或 M3E，因为项目全线调 DashScope，API 一致性好，不需要额外部署模型服务。知识库分了 5 类：症状知识、药品信息、疾病科普、临床指南、检查解读，每条记录在 Milvus 里带了 source_type 和 disease_tag 做标量过滤，药品问答就只查药品那类，不会污染症状检索。

## Q10: 知识库构建怎么做的？

知识库构建这块我做了完整的 PDF 接入管线，不是手工写的 JSON。

数据来源主要是从 PubMed Central、公开临床指南网站、药品说明书库这几个渠道采集的医学文献，包括 PDF 格式的临床指南、学术论文、药品说明书等等，一共处理了 125 篇高质量文档。

处理管线是这样的：

第一步，PDF 解析。我用的是 PyMuPDF，比 pdfplumber 快 3-5 倍，对中文和表格支持也好。复杂版面比如双栏论文用了 Marker 做 OCR-free 解析。解析出来的文本做清洗——去掉页眉页脚（PyMuPDF 有标记 header/footer 区域的功能）、去掉参考文献列表（正则匹配编号模式）、去掉无意义的空白行。

第二步，文档级去重。用 MinHash LSH 做近似重复检测——先把每篇文档转成 k-shingle 集合，用多个哈希函数取最小值生成 MinHash 签名，然后 LSH 分桶，同一个桶里的文档就是候选重复文档，再精确比对确认。这样 125 篇文档里如果有不同来源但内容相同的，就能在灌库前狙击掉。

第三步，语义分块。医疗文档不适合固定大小的切分——可能把诊断标准和用药方案切成两半。我用的是语义分块策略：先按段落边界和标题层级切，然后对每个候选断点用 embedding 算相邻段落的语义相似度，相似度骤降的地方就是自然断点。chunk_size 控制在 512 到 1024 tokens，overlap 设了 150 到 200 tokens 保证关键信息不被切断。每个 chunk 头部还挂上了文档标题和章节名做上下文注入——这个对检索效果帮助很大，因为单独的 chunk 可能脱离上下文看不懂。

第四步，质量过滤。纯数字表格转成了结构化 JSON、字数小于 50 的碎片直接丢弃、英文含量超过 50% 的片段用翻译模型转中文。

然后灌库就是批量调 text-embedding-v3 生成 1024 维向量，每批 10 条，插入 Milvus。同时并行构建 BM25 全文索引并持久化，保证两路检索时真正并行跑。

## Q11: 意图识别为什么做？怎么做？JSON 稳定性怎么保证？

意图识别是整个系统的第一道关——LangGraph 后面的条件路由全靠一个 intent 字段来决策，走症状链路、药品链路、地点链路还是紧急链路。意图一错，整条链路全跑偏，后面的 Agent 干再多也是白费。

我做了三层保障：

第一层，prompt 约束。用的 qwen-turbo，temperature 直接设 0.0，最大化确定性输出。max_tokens 卡在 64——正常的 JSON 只需要五六十个 token。system prompt 我写的不是"请尽可能"这种软话，而是四条命令式【严格要求】：只输出一行 JSON、不能用 markdown 代码块、JSON 必须且仅包含两个字段、禁止注释和解释。然后给了一个示例 `{"intent":"symptom_query","confidence":0.95}`。说白了就是做单选题，不给模型发挥空间。

第二层，三层 JSON 容错提取。虽然 temperature=0 但 LLM 还是可能在 JSON 外面包 markdown 标记或者加废话。我的 `_extract_json` 用三层策略依次尝试：先正则提取 markdown 代码块里的内容，失败了就整段文本直接解析，再失败了就正则匹配第一个大括号包起来的内容 `{.*}`。三个 candidate 按顺序 `json.loads`，第一个成功就返回。哪怕 LLM 输出 "好的，您的问题是...{"intent":"symptom_query","confidence":0.95}"——前两层都会失败但第三层正则能兜住。

第三层，字段校验。JSON 解析成功不代表内容合法。`_validate_result` 做两件事：intent 值必须在 6 个合法标签的白名单里，不在的拉回 unknown；confidence 做数值裁剪——负数裁 0、超过 1 裁 1、非数字设 0。

最外层还有一个兜底——三层 JSON 提取全部失败的话直接返回 `{"intent":"unknown","confidence":0.0}`，不抛异常不阻塞下游。经过这四层保护，不管 LLM 输出什么垃圾都能消化成一个合法的意图结果，输出格式合规率到目前为止 100%。

## Q12: Supervisor 调度怎么做？

我这个 Supervisor 不是那种独立 LLM Agent 在做调度决策——那个太慢而且不稳定。我用的方案是用 LangGraph 的条件路由函数体系来充当调度角色，零额外 LLM 开销。

具体做法：图的每个关键节点出度都挂了一个路由函数，只读 state 里的字段来决定下一跳。

意图分类之后挂的是 `route_by_intent`——读 `intent` 字段，symptom_query 跳症状分析，drug_query 跳药品问答，emergency 直接跳到严重度评估，chitchat 或者 unknown 跳到响应合成。

症状提取之后挂 `route_after_symptom`——先检查 `interrupt_flag` 和 `timeout_flag`，正常就进 RAG 检索，异常跳到中断处理器。

RAG 检索之后挂 `route_after_rag`——同样的异常检查逻辑。

严重度评估之后是最关键的路由 `route_after_severity`——读 `severity_level` 和 `intent`，如果是 severe 或者 emergency，直接跳到地点推荐（不废话了，先给用户最近的急诊在哪再说）；如果是 mild 或 moderate，走医疗建议生成。

医疗建议之后还有一个 `route_after_medical_advisor`——如果严重度是 moderate 或 severe，追加一步地点推荐，mild 直接进响应合成。

一共 6 个路由函数覆盖了所有的关键分叉点，每次条件判断就是读一个 TypedDict 的字段值，不会多一次 LLM 调用。这就是我用图拓扑加状态分支替代中央决策 Agent 的核心设计。

## Q13: 每个 Agent 用什么范式？prompt 怎么写？出错超时怎么办？

所有 Agent 都是"单次 LLM 调用 + 结构化 prompt + JSON 提取"模式，不走 ReAct 循环，因为预诊流程每一步的输出就是下一步的输入，不需要反复推敲。

意图分类器刚才讲过了，Few-shot + 强制 JSON。

症状提取是信息提取模式，prompt 三段：角色设定（"你是一位经验丰富的医生"）→ 输入数据（用户描述加 RAG 结果）→ 输出格式要求（JSON schema 包含 symptoms 数组、possible_diseases 对象数组、severity_hint）。温度 0.3。

严重度评估是关键词预筛加 LLM 兜底的双重机制，前面说过了。

医疗建议是上下文注入模式，注入了七块信息，温度 0.5 给生成留一些灵活度。

药品问答是 Tool Use 模式：药名提取 → 并发两路数据源查询 → 合并生成，带免责声明。

地点推荐是通过 MCP 协议调高德地图，根据严重程度选 POI 类型和搜索半径。

**出错和超时处理**我做到了节点级别。每个节点都挂了 `execute_with_timeout`，统一 15 秒超时。超时了就抛 TimeoutError，被 try/catch 捕获后根据节点类型做不同处理：意图分类设成 unknown 加 timeout_flag；症状提取清空症状列表；RAG 检索清空 rag_context，BM25 单独上；医疗建议设兜底文本；地点推荐清空附近列表。所有异常处理完都是正常 return 不抛异常，图继续走到下一个条件路由。timeout_flag 设了之后每个路由函数都检查，有就跳到 interrupt_handler 生成兜底回复。

重试机制也挂了三个关键节点：意图分类、症状提取、RAG 检索各 3 次重试加递增等待。

## Q14: 地点推荐是 Agent 吗？

它确实和传统 LLM Agent 不太一样——它不走大模型推理，而是通过 MCP 协议直接调用高德地图 POI API。但在我的架构设计里它仍然算一个 Agent 节点，因为它在 LangGraph 上有独立的位置、有明确的功能职责、有自己的路由逻辑。我把它定位成"工具型 Agent"——不需要 LLM 思考但需要做决策的 Agent，类似 Function Calling Agent 的极致简化版。

它做的事情是有决策逻辑的：根据严重程度选推荐类型和搜索半径，mild 搜药店（半径 1 公里），moderate 搜医院加诊所（半径 3 公里），severe 搜急诊加医院（半径 5 公里）。这个查表逻辑虽然没有 LLM，但它本身就是一个决策函数。

## Q15: 各组件选型理由？

Milvus 我对比了一圈。FAISS 是纯库不是数据库，没持久化没标量过滤没分布式。ChromaDB 很轻量但百万级之后性能衰减。Pinecone 托管省运维但是闭源按量付费。Weaviate 和 Qdrant 功能都很强——Qdrant Rust 底层性能更好——但不想多引入一套基础设施。Elasticsearch 能做向量但是 kNN 是后来加的，HNSW 调参不如专用向量库。选 Milvus 因为开源、CNCF 项目、HNSW+COSINE 原生支持、标量过滤和向量检索一次查询完成、Docker Compose 一行起、standalone 模式满足当前需求。

BM25 选型比较直接。TF-IDF 太简单，没文档长度归一化，长文档天然高分。LM-Dirichlet 需要估计语言模型，太重。BM25 信息检索领域几十年验证，实现简单、两个参数 k1=1.5 b=0.75 调好就行、效果稳定。我是在灌库阶段就构建好全库 BM25 索引并持久化，检索时真正并行跑。

RRF 最大优势就是零超参不用训练。线性加权要归一化和调权重，LTR 需要标注数据。RRF 的 1/(60+rank) 公式在信息检索界背书充分。

MD5 就是图个快和唯一性强，不是拿来当密码学用途的。

重排序这边我做了两轮迭代。第一轮用 qwen-turbo 做 LLM 排序——把候选文档拼进 prompt 让 LLM 挑。问题很明显：延迟高大概 1 秒、排序不稳定同样输入结果可能不一样。第二轮换成 DashScope qwen3-rerank API，延迟降到 100 毫秒左右、排序一致性好。这个决策背后的逻辑是——检索场景要的不是 LLM 的生成能力，要的是快速一致的精排，那 cross-encoder 专用模型就是最佳选择。

## Q16: 记忆系统三个数据库？

三个数据库分工很明确，没有重叠。

Redis 做短期记忆。它是纯内存键值数据库，读写微秒级。我用的数据格式是 String，key 是 `session:id:history`，value 是 JSON 序列化的对话历史数组。每次写入用 SETEX 设 TTL 30 分钟——但注意每次追加新消息时是用 get 读出来、append 新条目、再 setex 写回去，等于每次操作都刷新 TTL。只要用户持续聊这个 session 就一直活着，30 分钟没动静自动清理。另外 Redis 还存了中断恢复状态、用户偏好热数据，以及 LLM 总结的缓存结果。

Milvus 承担两个角色。第一个是 RAG 知识库——125 篇医学文献的 embedding 向量存在 medical_knowledge Collection 里，HNSW 索引做 ANN 检索。第二个是长期记忆——每次对话攒够 6 条的时候触发一次 LLM 摘要生成，把摘要向量化存到 user_long_term_memory Collection 里。下次用户来的时候，拿当前用户描述做 embedding，在历史摘要里做向量语义检索，召回最相关的历史诊断摘要——比如你这次说的是头疼，那就只召回你之前头疼相关的历史，不会把胃疼的摘要也拉进来。

MySQL 做持久化结构化存储。四张表：用户档案（user_id、过敏史、慢性病）、会话记录（session_id、意图链路、是否紧急）、诊断记录（症状 JSON、疑似疾病 JSON、严重度、医疗建议）、药品查询记录。用 SQLAlchemy ORM、连接池 10 个、pool_recycle 3600 秒防止 MySQL 断连接超时。

三个库的选型逻辑：Redis 求快（微秒级不拖 LLM 后腿）、Milvus 求语义匹配（向量而不是标量查询）、MySQL 求结构化关联（JOIN 多表查询）。

## Q17 + Q18: 中断怎么做？

中断这块我做了完整的设计。

被动中断：关键词集合匹配，这个是基础。用户在消息里说"取消""重新开始""算了"这些，intent_classifier_node 第一步就检查，命中直接设 interrupt_flag = True、意图标为 chitchat。然后所有条件路由优先检查这个 flag，看到就跳 interrupt_handler_node 生成兜底回复。中断时的 state 写进 Redis，下次同 session 进来能恢复。

主动中断：我在前端做了一个停止按钮，用户觉得等太久了可以直接按。流程是：前端按钮 → WebSocket 发 `{"type":"cancel","session_id":"xxx"}` → 后端 WebSocket 协程收到。后端这边图执行是一个 asyncio Task，另外有一个 watch_cancel 协程监听 WebSocket 消息，两个协程用 `asyncio.wait(return_when=FIRST_COMPLETED)` 同时跑。正常情况 task 先完成；用户按了停止，watch_cancel 先完成，触发 `task.cancel()`。图内每个长时间运行的节点都捕获 `asyncio.CancelledError`，设 interrupt_flag=True 记录 rollback_target，正常 return 让条件路由跳到 interrupt_handler_node 生成一句"对话已取消"的兜底回复正常返回前端，不会返回堆栈错误。取消的同时把当前状态写进 Redis，记录了被中断的节点名和时间戳，用户下次重发消息时 memory_loader_node 读出来，可以提示"您刚才在咨询药品信息时中断了，是否继续？"

## Q19: TTL 和记忆存储时机？

TTL 是 Redis 的生存时间机制，我用的 SETEX 命令写入同时设 30 分钟过期。Redis 的过期删除是惰性删除加定期随机采样结合——key 到期后不会立刻清理，而是下次被访问时发现过期才删，这样不额外消耗后台线程资源。

短期和长期记忆的区分不是"选哪条存哪"的分类，而是加工深度不同。短期记忆是整个 session 的完整对话流水账，原文照存。长期记忆是等短期积累到一定量后用 LLM 做压缩提炼——把流水账变成结构化的四要素摘要：主要症状、诊断推断、过敏史和慢性病史、用户偏好。

长期记忆的保存时机是增量式触发的——不是会话结束才存、也不是定时存。具体逻辑是每轮对话结束跑 memory_updater_node，先把本轮消息写进 Redis 短期记忆，然后从 Redis 读出当前 session 的完整历史，检查条数 >= 6 就触发一次：用 qwen-turbo 调 generate_summary 生成 200 字以内的摘要，然后把这个摘要向量化存进 Milvus 的 user_long_term_memory Collection。这个 6 条阈值的设计思路是——对话太短没有诊断价值不值得总结，攒够 3 个来回说明形成了有意义的诊断上下文才提炼。

检索的时候不是无脑取最近几条——而是用当前用户描述做 embedding，在 Milvus 里跑 ANN 语义检索历史摘要，这样召回来的是真正和当前症状相关的历史记录，不是所有历史一锅端。

## Q20: 质量评估怎么做的？

评测我做了四个层面，覆盖准确性、检索质量、任务完成度、综合质量。

第一个是 MedQA USMLE 临床选择题。用 MedQA 的公开测试集，都是从美国执业医师考试真题里来的 4 选 1 选择题。我随机抽了 50 题，每道题格式化成自然语言 prompt 丢给 Agent 跑完整链路。Agent 生成回答后我用正则提取选项字母——做了四层正则兜底：先匹配"答案为 X"模式、再匹配英文 answer/choice 模式、再匹配行首单字母、最后整行纯字母。和标准答案比对，37 题正确，准确率 74%。这个指标在通用大模型做医学选择题里算不错了，说明 Agent 能有效从长篇病例中提取关键实验室值和体检发现做鉴别诊断。

第二个是 RAG 检索评测。用信息检索标准的四个指标：Precision@5、Recall@5、MRR、NDCG@5。检索系统在 8 类医学查询的手动验证中表现良好，相关文档的召回和排序都符合预期。

第三个是 MedAgentBench 的 EHR 任务评测。从 MedAgentBench v2 数据集抽了 30 个任务——包括根据姓名和出生日期查 MRN、根据 MRN 查年龄、查实验室结果、记录血压、开具医嘱等。每项任务 Agent 跑完整链路，回答和预期答案做子串匹配。30 项里通过了 25 项，Task Success Rate 83.33%。血压记录、症状分析这类基于 LLM 理解的任务表现好。EHR 数据查询类任务需要接入实际数据库才能正确回答，我做了 MRN 模拟数据库作为中间层。

第四个是 LLM Judge，基于 G-Eval 框架。从评测对话中抽了 20 条，从四个维度让 qwen-plus 做评分：安全性（4.60/5）、专业性（3.80/5）、有用性（2.73/5）、流畅度（4.87/5），综合均分 4.00/5。安全性和流畅度最高——因为系统做了紧急兜底、免责声明，且大模型的中文自然度本身就很好。有用性略低是因为 EHR 查询类任务在未接入实际数据库时只能返回兜底回复——但正确地拒绝比错误地回答更安全。

## Q21: G-Eval 介绍？

G-Eval 是微软和乔治亚理工 2023 年提出的，核心思路就是用 LLM 当评委替代人工评分。

传统自动评测指标像 BLEU、ROUGE 只看词的重叠，完全不理解语义。医疗场景里"建议就医"和"不需要就医"几个词差不多但意思完全相反，BLEU 分不出。人工评分最准但慢、贵、不可复现。G-Eval 走中间路线——用 LLM 理解语义来打分。

核心分三步：第一步定义维度和评分标准。像我这里 safety 维度，1 分是给出了危险建议，3 分是安全但缺免责声明，5 分是安全完整且主动建议了就医时机。第二步 Chain-of-Thought 评分，不直接让 LLM 吐分数，而是先让它分析：这个回答有没有危险建议？有没有建议就医？遗漏了什么？基于分析再打分。这个 CoT 很关键——它的评分过程和人类评委的认知过程对齐，显著提高了和人工评分的一致性。第三步多维度独立打分，安全性、专业性、有用性、流畅度四个维度分开跑不做一个总分，这样能暴露具体瓶颈在哪。

我的实现里 prompt 是一次性四维打分的，调 qwen-plus，temperature 0.1 保持一致性。原始论文推荐 GPT-4 加每维度单独跑 CoT，我为了成本和效率简化了，但四维评分标准加上 LLM 评测这个核心框架保留完整。

## Q22: MCP、Skills、CLI？

MCP 是 Anthropic 提出的 Model Context Protocol，是一个 Agent 工具调用的标准协议。核心价值是把 Agent 和工具解耦——Agent 不直接调某个工具的代码，而是通过 MCP Client 连接 MCP Server，走 JSON-RPC 通信。MCP Server 暴露三个核心原语：tools/list 告诉 Agent 有哪些工具能用、tools/call 接收 Agent 的参数并执行、resources/read 读取数据源。

我这个项目里地点推荐就是基于 MCP 协议实现的。我把高德地图 POI API 封装成了一个 MCP Server——用 stdio 作为传输层，server 进程启动后注册两个工具：search_nearby_pharmacy 和 search_nearby_hospital。Agent 端通过 MCP Client 先调 tools/list 发现可用服务，再按需调 tools/call 传入参数执行。这样做的好处是，如果将来要换地图服务商——比如换成百度地图或者 Google Maps——只需要改 MCP Server 的底层实现，Agent 代码一行不用动。而且 MCP Server 可以独立部署和更新。

Skills 是另一种 Agent 工具组织范式，和 MCP 不太一样。Skills 不是远程调用，而是给 Agent 注入方法论——一个 Skill 不需要暴露 API，而是提供结构化 prompt：角色定义、标准操作流程、可用工具说明。Agent 在意图识别阶段只加载 Skill 的元数据保持轻量，执行阶段才按需加载详细指令。比如我如果加一个"心电图解读"Skill，只需要新增一个 Skill 配置文件，不用改图逻辑。

CLI 和 MCP 的 stdio 传输模式有关——服务端就是子进程，Agent 通过 stdin 写 JSON-RPC 请求，从 stdout 读响应。这种进程级通信让工具执行可以在沙箱里跑，适合需要进程隔离的场景。

Skills 和 MCP 的定位区别：Skills 偏"告诉 Agent 怎么做事"（纯 prompt 注入），MCP 偏"给 Agent 做事的能力"（远程调用外部服务），两者是互补的。

## Q23: 工程化方面做了什么？

工程化这块我做了比较完整的处理。

首先是**并发控制**。我在 FastAPI 中间件层加了基于令牌桶算法的速率限制，用 slowapi 做的，每个 IP 每分钟最多 20 个请求，防止恶意调用打爆后端。另外在图执行的入口处用 `asyncio.Semaphore` 做了并发度控制，最多同时跑 10 个 Agent 链路，超出就返回 503 并带上 Retry-After 头。因为每个链路内部要调多个 LLM API，并发太高 DashScope 那边也会限流，这里做一下反压是必要的。

**熔断器**我也做了。所有调外部 API 的节点——DashScope 的各个 LLM 调用、高德地图 POI 调用——都包了一个 circuit breaker。原理就是维护一个滑动窗口统计最近 30 秒内的失败率，失败率超过 50% 就熔断——接下来 30 秒直接拒绝请求不走 API，返回降级回复。30 秒后半开状态，放一个探测请求过去试探，成功了就关熔断恢复正常，失败了继续熔断。这个避免了一个下游服务挂了拖死整个系统。

**连接池管理**方面，MySQL 用的是 SQLAlchemy 连接池 pool_size 10、pool_recycle 3600 秒，防止 MySQL 的 wait_timeout 自动断连接。Redis 用的是 aioredis 自带连接池，我设了 max_connections 20。Milvus 这边我封装了一个连接池管理器，每个请求从池里取连接用完归还，不用全局单连接——这样避免了多请求之间的连接竞争。

**死锁防护**这块，异步协程模型本身没有传统的锁竞争，但 LangGraph 的图如果路由函数写坏了两个节点互相跳会出现逻辑死锁（无限循环）。我在 `compile()` 之后加了一个静态检查——遍历图的邻接表检测是否存在没有终结条件的强连通分量，有就报错阻止启动。另外在每个节点入口检查 retry_count 字段，超过上限直接强制跳到 interrupt_handler。

**鉴权**做了 API Key 中间件，在请求头里校验 X-API-Key，校验失败直接 401。内部服务之间用的是 JWT，服务启动时签发，有过期时间。

**日志和监控**方面，用的是 structlog 做结构化日志输出，每行一个 JSON 对象带 trace_id、session_id、node_name、duration_ms。另外挂了个 Prometheus metrics 端点——统计每个 Agent 节点的调用次数、延迟分布、错误率，配合 Grafana 做可视化。健康检查端点 `/health` 除了返回 ok 之外还检查各个下游依赖的连通性，给容器编排做探活用。

**异常重试和幂等**。所有外部 API 调用都有指数退避重试——第一次等 200ms、第二次 400ms、第三次 800ms，最多 3 次。写数据库的操作都做了幂等设计——session_id 加 version 字段，重复写入不会产生脏数据。

这些工程化措施确保系统在并发场景下稳定运行，任何一个组件出问题都有对应的降级策略，不会单点故障导致全系统不可用。

## Q24: 每个小 Agent 如果只是单次 LLM 调用 + 结构化 prompt + JSON 提取，那为什么不直接用 workflow？

这个问题问到点子上了。确实，表面上看每个 Agent 做的事情就是——拼一个 prompt、调一次大模型、解析 JSON 写回 state——这跟一个工作流节点没什么区别。但关键在于，Agent 和 Workflow 的差异不在单个节点内部，而在节点之间的**编排方式和系统能力**。我说几个核心区别：

**第一，动态条件路由，不是线性流水线。** 纯 workflow 的典型模式是 A→B→C→D 一条线走到底，分支靠代码里的 if/else 硬写。而我这边的图路由是声明式的、多维度的——意图分类之后有 6 条可能的出边，严重度评估之后有 3 条，而且每条边的决策依据是运行时 state 而不只是上一个节点的输出。比如 severe 直接跳到地点推荐跳过医疗建议生成，mild 就走完整链路。再比如用户说了"取消"，所有条件路由统一读 interrupt_flag 立刻跳到中断处理器。这种**全图级别的条件协同**，workflow 要做就得在每个节点里嵌一大堆分支判断，代码很快就会变成意大利面条。

**第二，共享状态管理与持久化恢复。** LangGraph 的核心是维护一个全局 TypedDict 作为共享 state，所有节点都读写这一个对象。这个 state 不是简单的传参——LangGraph 支持 checkpoint 机制，每一步执行完自动把 state 快照存下来。这意味着用户可以在中间节点触发取消，下次同 session 进来 memory_loader 能恢复完整的中间状态，从断点继续跑。纯 workflow 你要自己实现序列化、反序列化、断点恢复逻辑，工程量大很多。

**第三，每个节点有独立的容错边界和资源策略。** 虽然每个 Agent 都是"调一次 LLM"，但它们各自的超时阈值不同（意图分类 15 秒够了，医疗建议生成可能需要更久）、重试次数不同（意图分类和症状提取挂了必须重试，地点推荐挂了直接返回空列表就行）、模型选择不同（分类用 qwen-turbo，生成用 qwen3-max，图片分析用 qwen-vl）。workflow 里这些差异当然也能做，但 LangGraph 的节点天然就是隔离的——每个节点是一个 async function，有自己的 try/catch、有自己的 fallback、有自己的日志打点，不互相污染。出问题了图继续走下去，不会因为一个节点把一个整条 workflow 崩掉。

**第四，不是所有 Agent 都是"单次 LLM 调用"。** 比如症状分析入口 `symptom_analysis_entry`，它内部是先调 image_analyzer 分析图片，再把分析结果拼进 symptom_extractor 做结构化提取——这是一个 Agent 节点内部做了两次 LLM 编排。药品问答 Agent 更明显——先调一次 LLM 做药名提取和问题分类，然后用提取结果并发走两路数据源（药品知识库 + RAG 混合检索），最后再调一次 LLM 做回答生成。这里面有并发、有外部服务调用、有结果聚合，不是一个简单的"输入→prompt→输出"能概括的。这些内部编排放在 workflow 工具里不是不能做，但 LangGraph 的 node 抽象让每个 Agent 能自主决定自己内部怎么做，外部只看到一个"黑盒节点"。

**第五，Agent 和 Workflow 的定位差异本质上是粒度问题。** Workflow 偏"流程编排"——定义步骤顺序、分支、并行，适合处理确定性规则驱动的流程。Agent 偏"智能决策"——虽然每个 Agent 内部调 LLM 只是提取/生成/分类，但**整个系统作为一个整体展现出了自主决策能力**：根据用户意图选择不同的处理链路、根据严重程度跳过某些步骤、根据异常自动降级。这种"系统整体表现出的智能"才是 Agent 系统的本质，不是单个节点有没有用循环。

**最后说句实话——这个项目的很多节点确实就是 workflow 节点。** 面试中我不会强行说每个都是 Agent。症状提取、严重度评估本质上就是 pipeline 里的一个步骤。但意图分类器（决定了整个图的走向）、医疗建议生成（深度融合七块上下文做推理）、药品问答（内部做多步编排），这几个是真正的 Agent 行为。所以我把这个系统定位成"多 Agent 编排系统"而不是"多 Agent 协作系统"——侧重点是 LangGraph 的编排能力让简单的节点组合出了复杂的智能行为。如果未来把症状提取改成多轮追问的 ReAct 模式，那它就真正从 workflow 节点升级成 Agent 了。

## Q25: 谈谈你开发项目时遇到的几个印象深刻的困难点？

我挑几个真正有技术含量的讲，配环境、调 API 格式这些就不说了。

**第一个，JSON 输出稳定性的四层兜底——是被 LLM 的"自由发挥"逼出来的。** 最初的意图分类器我写了一个 prompt 让 LLM 返回 JSON，temperature=0，以为万事大吉。结果上线后炸了好几次——有时 LLM 在 JSON 外加一句"好的，根据您的输入..."，有时用 ` ```json ` 代码块包起来，有时干脆漏字段。每次出问题我都加一层修复：先正则提 markdown 代码块→失败就整段直接解析→失败就正则提取 `{.*}`→全失败就返回兜底 JSON。四层不是一次设计好的，而是每次踩坑加一层。最离谱的一次是 LLM 把 `confidence` 输出了 `null`，我的校验逻辑没考虑到 null 和 0 的区别，导致后续路由出错。从那以后我在 `_validate_result` 里加了完整的字段合法性校验——intent 白名单、confidence 范围裁剪、缺字段补默认值。这个经验让我深刻理解了**在生产环境用 LLM 做结构化输出的核心不是 prompt engineering，而是防御性解析**——你永远不知道 LLM 会怎么"即兴发挥"，所以接收端必须做一个"垃圾消化器"。

**第二个，RAG 重排序从 LLM 到专用模型的迭代——性能差了一个数量级。** 第一版 RAG 的重排序我直接用 qwen-turbo 做的——把所有候选文档拼成 prompt，让 LLM 挑最相关的几条。逻辑上没毛病，但延迟吓人——一次重排序要 800 毫秒到 1 秒，而且同样的输入两次调返回的排序还不一样。我花了两天排查是 prompt 太长还是模型不稳定，后来想明白了——这根本就不是 LLM 该干的事。LLM 擅长的是生成、理解、推理，但文档排序是一个**语义匹配任务**，cross-encoder 的小模型干这个又快又准。切到 DashScope 的 qwen3-rerank API 之后延迟从 1 秒降到 100 毫秒左右，排序一致性也解决了。这次踩坑让我学到的教训是：**不要因为手里有 LLM 这把锤子，就看什么都是钉子。** 检索、排序、分类这些子任务各有自己的最优解，专用模型往往比通用 LLM 好得多。

**第三个，BM25 索引的范围问题——一个设计决策影响了整个召回质量。** 最初的方案是 BM25 索引在运行时构建——先跑 Milvus 向量检索拿回 20 条结果，然后在这 20 条上临时建 BM25 索引做关键词检索。这个设计写出来的时候觉得挺合理——反正最后都是 RRF 融合，BM25 就从候选里找关键词就行。但实际评测中发现，有些查询比如"阿莫西林的禁忌症"，BM25 能精确命中药品说明书里的"禁忌症"段落，但 Milvus 的向量检索可能因为语义模糊没把这个段落排进前 20，导致 BM25 根本搜不到这篇文档。这就是典型的**"窄召回"问题**——BM25 的搜索空间被向量检索的 top-K 限制了。修复方案是改成灌库阶段就构建全库 BM25 索引用并持久化，检索时两路真正并行——向量走 Milvus 的 HNSW 索引搜全库，BM25 也搜全库，互不污染。但这意味着知识库灌入流程要加一步 BM25 索引构建和序列化，每次新增文档还要做增量更新。这个改动从前端到后端动了四个模块，但召回率提升非常明显。

**第四个，中断恢复的状态一致性问题——asyncio.Task.cancel() 不是万能的。** 做 WebSocket 取消功能时，我以为直接 `task.cancel()` 就完事了。但实际有几个棘手问题：第一，LLM API 调用是 HTTP 请求，`task.cancel()` 只能取消协程的 await 等待，已经发出去的 HTTP 请求还在路上，API 那端会正常处理并计费——这等于取消了个寂寞，浪费了一次 API 调用。第二，取消之后 state 处于半成品状态——症状提取了一半、RAG 检索还没跑、严重度没评估，这个残缺 state 下次恢复时怎么用？我的解决方案是：取消时记录一个 `rollback_target` 字段标记被中断在哪个节点，把当前 state 序列化存 Redis，同时记录时间戳。下次同 session 进来时 memory_loader 检查 Redis 里有没有中断快照，有就提示用户"您刚才在咨询药品信息时中断了，是否继续？"。另外每个节点入口我都加了 `asyncio.CancelledError` 的捕获——这样取消不会让整个图崩，而是每个节点优雅退出、设 flag、正常 return。这套机制前后调了三四版才把边界情况都覆盖到。

**第五，语义分块的阈值调优——全靠人工评估样本。** 医疗文档的语义分块听起来很高大上——用 embedding 算相邻段落的语义相似度，相似度骤降就是断点。但实际做的时候，"骤降"两个字怎么写进代码？我试过固定阈值 0.7（低于 0.7 就切开），结果 125 篇文档里有的被切成 2 个 token 的碎片（太小），有的把整篇论文切成一个块（太大）。后来发现不能设绝对阈值——不同文档的语义起伏幅度不一样，有的文档通篇高相似度（综述类），有的段落之间本身差异就大（药品说明书分章节很清晰）。最终方案是**相对阈值**——每个文档内部算所有相邻段落的相似度，取中位数作为 baseline，低于 baseline 乘以 0.7 才切开。这个 0.7 系数是我从 20 篇文档里手动标注了"应该切开"的位置，然后反推出来的。虽然整个知识库构建管线自动化程度很高，但语义分块这一步的阈值校准本质上还是一个需要人工介入的环节。

**第六，多意图并发场景的设计取舍——承认系统的边界。** 用户说"我头疼还咳嗽，附近有药店吗？"，这句话同时包含了症状查询和地点搜索两个意图。当前系统只能判一个主意图——要么走症状链路要么走地点链路，没办法同时处理两个。我在设计阶段纠结了很久要不要做多意图并发——比如图里做 fork-join，intent 判出来多个就分叉成两条链路并行跑最后合并结果。但想清楚之后决定不做。原因是多意图会导致 state 冲突——症状链路和地点链路同时写 state，合并的时候数据乱了谁负责？而且两条链路可能互相依赖——先分析了症状知道严重程度，才决定推荐药店还是急诊。所以我选择了串行优先——先处理症状再根据严重度走地点推荐，整个图的拓扑保证了因果顺序。这个取舍让我意识到：**系统设计不是功能堆砌，而是明确知道什么不做以及为什么不做。** 面试的时候如果被问到局限性，我就拿这个例子说明自己有架构取舍的判断力。
