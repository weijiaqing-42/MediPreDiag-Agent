# MediPreDiag-Agent 面试问答记录（原版）

---

## Q1: 简单介绍一下你这个项目吧，然后整体的架构是怎么样的？

**A:** 这是一个面向医疗预诊场景的多智能体协作系统。用户用自然语言描述症状，系统通过多个专职 Agent 协同工作，给出症状分析、严重程度评估、药品建议和就医推荐。

整体架构分五层：

**接入层** — FastAPI 提供 REST 和 WebSocket 双通道，WebSocket 用于流式推送 Agent 各节点的中间结果，前端是 HTML/CSS/JS 单页应用。

**编排层** — LangGraph 有向图驱动。入口统一走 `memory_loader` 加载会话上下文，然后是 `intent_classifier` 做意图分类。分类结果通过条件边路由到不同 Agent 节点。

**Agent 层** — 5 个真正含 LLM 推理的 Agent：意图分类器（qwen-turbo Few-shot）、症状提取（qwen3-max 结构化提取）、严重度评估（关键词预筛 + LLM 分级）、医疗建议生成（融合 RAG 结果）、药品问答（药名提取 → 知识库查询 → 生成）。外加 RAG 检索和地点推荐两个 Tool 节点。

**记忆与存储层** — Redis 短期会话（TTL 30分钟）、Milvus 长期记忆（诊断摘要向量化）、MySQL 持久化。

**检索层** — 向量 + BM25 双路并行 → RRF 融合 → MD5 去重 → DashScope qwen3-rerank 重排序。

Docker Compose 一键部署，底层依赖 MySQL、Redis、Milvus、etcd、MinIO 五个容器。

## Q2: 先从接入层开始，对 FastAPI 了解吗？REST 和 WebSocket 呢？具体做了什么？

**A:** FastAPI 是基于 Starlette 的异步 Python Web 框架，核心卖点是自动生成 OpenAPI 文档、Pydantic 类型校验、原生 async/await 支持。

接入层做了三件事：

**第一，生命周期管理。** 用 `lifespan` 异步上下文管理器，启动时按序初始化 Redis → Milvus → MySQL，每个包 try/catch 和超时控制，一个挂了不影响其他，服务降级启动。

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    try:
        await asyncio.wait_for(redis_client.connect(), timeout=5.0)
    except Exception as e:
        print(f"Warning: Redis not available: {e}")
    try:
        milvus_client.init_collection()
    except Exception as e:
        print(f"Warning: Milvus not available: {e}")
    try:
        init_mysql_schema()
    except Exception as e:
        print(f"Warning: MySQL not available: {e}")
    yield
```

**第二，REST 端点** (`POST /api/v1/chat`)。Pydantic `ChatRequest` 校验 → 展开成 `MediState` → `graph.ainvoke` 执行 → 映射到 `ChatResponse` 返回。

**第三，WebSocket 端点** (`/ws/chat`)。全双工协议，Agent 图谱多步执行时先回 `status: Processing`，最终结果 `type: response` 推送。

**为什么双通道？** REST 适合集成调用，WebSocket 适合前端交互——用户等待时看到状态变化。

## Q3: 降级启动是怎么设计的？超时了组件挂了怎么办？

**A:** 设计上有处理到位的也有盲区。

**已处理的——启动层：** lifespan 里每个初始化包 try/catch，Redis 5 秒超时。挂了打 Warning，服务照样起。

**已处理的——运行时记忆读写：** memory_nodes 里每次 Redis/Milvus 操作都包 try/catch，加载失败不影响主流程。

```python
# memory_nodes.py 典型模式
try:
    history = await short_term_memory.load(session_id)
    state["short_term_history"] = history
except Exception as e:
    logger.error(f"Short-term memory load failed: {e}")
```

**没覆盖的盲区：** Milvus 运行时挂了，RAG 检索的 `milvus_client.search()` 没有额外 try/catch 包装，异常会穿透导致 HTTP 500。

## Q4: 简历里的业务中断/回滚/超时降级机制体现在哪？

**A:** 诚实说：有落地的，有只搭框架没跑通的。

**重试 — ✅ 部分。** `execute_with_retry` 实现了 3 次重试 + 递增等待（1s→2s→3s），挂在了 `intent_classifier_node` 的 `classify_intent` 调用上。

```python
result = await interrupt_handler.execute_with_retry(classify_intent, user_message)
```

**中断 — ✅ 检测+路由通了。** 关键词集合 `{"取消","重新开始","算了","不用了","退出","停止"}` 匹配用户输入，设 `interrupt_flag = True`，每个条件路由函数都检查这个 flag。但只在消息开始时生效。

**超时 — ⚠️ 结构有，未接入。** `execute_with_timeout` 方法写了但没任何节点调用它。

**回滚 — ❌ 没有。** `rollback_target` 字段定义了但从未被赋值。

## Q5: 编排层用的 LangGraph，介绍一下。还了解哪些？

**A:** LangGraph 是 LangChain 团队的 Agent 编排框架，把工作流建模成有向图。每个节点是函数，边分普通边（固定流转）和条件边（运行时动态选择）。底层是状态图，维护共享 TypedDict 做状态。

项目里用法：`StateGraph(MediState)` 建图 → `add_node` 挂 10 个节点 → `add_conditional_edges` 做 6 个条件路由 → `compile()`。

也了解 CrewAI（角色扮演式，概念直观但不支持复杂分叉）、AutoGen（对话驱动，微软生态）、MetaGPT（SOP模拟，太重）、Dify/Coze（低代码平台）、手写编排（轻量但无持久化）。选 LangGraph 因为医疗流程天然多分叉，条件路由是其强项。

## Q6: 为什么要用多智能体？每个智能体怎么设计的？

**A:** 两个原因：**职责单一 + 可控性更好**。单 Agent 容易多任务串扰，prompt 长且难调试。拆开后每个节点只干一件事，出问题能定位到具体 Agent。

设计都是"单节点函数"——每个 Agent 是异步函数，输入输出都是 `MediState`：

| Agent | 职责 | 模型 |
|--------|------|------|
| 意图分类器 | 6 分类 | qwen-turbo |
| 症状提取 | 结构化症状+疑似疾病 | qwen3-max |
| RAG 检索 | 召回医学知识 | text-embedding-v3 |
| 严重度评估 | mild/moderate/severe | qwen-turbo |
| 医疗建议 | 整合信息生成建议 | qwen3-max |
| 药品问答 | 药名提取+知识库搜索+回答 | qwen-turbo+qwen3-max |
| 地点推荐 | 调高德 POI API | 无 LLM |
| 响应合成 | 结果拼成自然语言 | 按 intent 分路 |

## Q7: Agent 范式了解吗？REACT 之类的。

**A:** 了解的。主要几类：

- **ReAct** — Thought→Action→Observation 循环，推理和行动交错
- **CoT** — 纯推理，"step by step" 分步输出
- **Plan-and-Execute** — 先规划再执行，静态计划
- **Function Calling** — 最主流，Schema 约束的工具调用
- **Reflection** — 自我检查+修正，进阶版 Reflexion 存长期记忆
- **Multi-Agent/Supervisor** — 多 Agent 各司其职统一调度
- **Tree/Graph of Thoughts** — 多候选路径探索+剪枝

我的项目接近 Multi-Agent + 条件路由混合，不走 ReAct 循环，因为预诊是流水线式流程。

## Q8: RAG 混合检索和重排序具体讲讲。

**A:** 四个阶段串行：

**第一阶段：双路并行。** Milvus HNSW 向量检索（text-embedding-v3, 1024维, COSINE, Top-20）+ BM25 稀疏检索（jieba 分词, Top-20）。

BM25 公式：IDF × TF × 文档长度归一化（k1=1.5, b=0.75）。

**第二阶段：RRF 融合。** `1/(k+rank)`，k=60，不关心分数绝对值只看排名。

```python
def _rrf_fusion(self, vector_results, bm25_results, top_k=30):
    k = 60
    scores = {}
    for rank, hit in enumerate(vector_results):
        scores[hit["id"]] = scores.get(hit["id"], 0) + 1.0/(k+rank+1)
    for rank, (bm25_idx, _) in enumerate(bm25_results):
        scores[f"bm25_{bm25_idx}"] = scores.get(f"bm25_{bm25_idx}", 0) + 1.0/(k+rank+1)
```

**第三阶段：Content MD5 去重。**

```python
content_hash = hashlib.md5(item.get("content", "").encode()).hexdigest()
if content_hash in seen_hashes:
    continue
seen_hashes.add(content_hash)
```

**第四阶段：DashScope qwen3-rerank 重排序。** 调 `/text-rerank` 接口做 cross-encoding，输出语义相关性分数。

选型踩坑：一开始用 qwen-turbo 做 LLM 排序，延迟 ~1s 且不稳定。换 Rerank API 后 ~100ms 稳定。

## Q9: RAG 链路为什么这么设计？

**A:** 漏斗式逐步收紧：宽召回（双路互补）→ 粗排（RRF 零成本）→ 精排（Rerank 高成本但少文档）→ 注入。

**为什么双路并行？** 向量保语义（"偏头痛"→"血管性头痛"），BM25 保关键词（"阿莫西林"精确命中），互补且并行不增延迟。

**为什么 RRF 而非线性加权？** 两种分数尺度不同无法直接相加，RRF 只看排名跨系统可比，零超参。

**为什么加重排序？** RRF 只给排名不告诉你语义多紧。cross-encoding 精确但贵，放漏斗末端只跑 30 条。

**不一轮解决的原因：** 125 篇全量 cross-encoding 不可接受。双路宽召回（低成本）→ 精排（高成本少量文档）是折中。

Embedding 选 text-embedding-v3（1024维）而非 BGE-m3：统一 DashScope 生态免额外部署。知识库 125 篇分 5 类，支持 source_type 标量过滤。

端到端延迟 ~200ms。

## Q10: 知识库构建怎么做的？chunksize、清洗过滤？

**A:** 诚实说——不是从 PDF 构建的。是 125 条手工整理的中文医学 JSON。`data/medical_knowledge.json`，三个字段：`source_type`、`disease_tag`、`content`。每条 300-800 字，本身就是独立知识片段，不需要 chunking。

灌库流程：读 JSON → 批量 embedding（每批 10 条）→ 插入 Milvus。

如果接 PDF 会怎么做：PyMuPDF 解析 → 语义分块（512-1024 tokens, 150-200 overlap）→ MinHash LSH 去重 → 质量过滤（字数<50 丢弃、纯数字表转 JSON）。

## Q11: 意图识别为什么做？怎么做？JSON 提取怎么保证稳定？

**A:** **为什么：** 意图是 LangGraph 条件路由的唯一决策依据，错了整个链路全跑偏。

**怎么做：** 三层保障——

**第一层，prompt 约束。** qwen-turbo，temperature=0.0，max_tokens=64，命令式 prompt + 输出格式示例。

**第二层，JSON 容错提取。** 三层递进：

```python
# 策略1：提取 markdown 代码块内容
code_block = re.search(r"```(?:json)?\s*(.*?)\s*```", stripped, re.DOTALL)
# 策略2：直接解析整段
candidates.append(stripped)
# 策略3：正则匹配第一个 {...}
brace_match = re.search(r"\{.*\}", stripped, re.DOTALL)
```

按顺序 `json.loads`，第一个成功就返回。

**第三层，字段校验。** intent 白名单检查（6 个合法标签），confidence 数值裁剪（0~1）。

最外层兜底：三层全失败返回 `{"intent":"unknown","confidence":0.0}`，不抛异常。

## Q12: Supervisor 调度怎么做？

**A:** 不是独立 LLM Agent，而是 **LangGraph 的条件路由函数体系**。每个关键节点出度挂路由函数，读 state 字段决定下一跳。

6 个路由函数：`route_by_intent`（读 intent）、`route_after_symptom`（检查 interrupt/timeout）、`route_after_rag`、`route_after_severity`（读 severity+intent）、`route_after_medical_advisor`（严重追加地点推荐）、`route_after_location`。

每次判断只读 TypedDict 字段值，无额外 LLM 开销。核心思路：用图拓扑 + 状态分支替代中央决策 Agent。

## Q13: 每个 Agent 用什么范式？prompt 怎么写的？出错怎么办？

**A:** 单次 LLM 调用 + 结构化 prompt + JSON 提取，不是 ReAct 循环。

- **意图分类器**: Few-shot, temperature=0.0, 命令式 prompt
- **症状提取**: 角色设定 → 输入数据 → JSON schema 输出, temperature=0.3
- **严重度评估**: 关键词预筛（19 个紧急词）+ LLM 兜底, temperature=0.1
- **医疗建议**: 七块上下文注入（症状、疾病、严重度、历史、长期摘要、RAG）, temperature=0.5
- **药品问答**: Tool Use 模式——药名提取 → DrugInfoLookupTool → RAG 补充 → 生成
- **地点推荐**: 纯工具调用——查策略字典 → 调高德 API

**出错处理：** 每个节点 try/catch，策略是让服务继续跑：意图失败→unknown，症状失败→清空，RAG 失败→空上下文，医疗建议失败→兜底文本，地点失败→空列表。超时没做（execute_with_timeout 未接入）。

## Q14: 地点推荐和 Agent 没关系吧？

**A:** 你说得对，不配叫 Agent。代码就是读 location+severity → 查策略字典 → 调高德 API，无 LLM。严格算是 Tool。

所以实际含 LLM 推理的 Agent 是 5 个：意图分类器、症状提取、严重度评估、医疗建议、药品问答。RAG 检索和地点推荐是 Tools。

## Q15: RAG 各组件选型？

**A:** 
- **Milvus 选型**: 对比 FAISS（无持久化）、ChromaDB（百万级衰减）、Pinecone（闭源付费）、Weaviate/Qdrant（功能强但多一套基建）、ES（kNN 后加的）。选 Milvus 因为开源+CNCF+HNSW+COSINE+标量过滤+Docker Compose 一行起。
- **BM25 选型**: 对比 TF-IDF（无长度归一化）、LM-Dirichlet（太重）。BM25 是最优折中。但有个实现取舍：BM25 索引每次检索时从向量结果即时构建，不是全库并行。
- **RRF 选型**: 对比线性加权（需归一化调权重）、LTR（需标注数据）。RRF 零超参、不用训练。
- **MD5**: 哈希唯一性保证，碰撞概率在 125 篇可忽略。
- **重排序选型**: 第一次迭代用 qwen-turbo LLM 排序（~1s, 不稳定），最终换 DashScope qwen3-rerank（~100ms, 稳定）。

## Q16: 记忆系统，三个数据库介绍。

**A:**
- **Redis**: 内存键值数据库，微秒级读写。存短期会话记忆（`session:{id}:history`, JSON 序列化对话历史, TTL 30min）+ 中断恢复状态。
- **Milvus**: 向量数据库，HNSW 索引。双角色：RAG 知识库（125 篇文献 embedding）+ 长期记忆（用户诊断摘要向量化）。
- **MySQL**: 关系型数据库。四张表：users、sessions、diagnoses、drug_queries，通过 user_id/session_id 关联。

选型原因：Redis 求快（微秒级不拖 LLM 后腿）、Milvus 求语义匹配、MySQL 求结构化关联查询。

## Q17: 中断怎么做？怎么判定？

**A:** 两大层面：

**中断检测：** 关键词集合 `{"取消","重新开始","算了","不用了","退出","停止"}`，在 `intent_classifier_node` 第一步匹配。命中设 `interrupt_flag=True`，条件路由跳到 `interrupt_handler_node` 生成兜底回复。

**中断恢复：** Redis 存 state（`session:{id}:interrupt`），下次同 session 消息进来读出来恢复。

**局限：** 只在消息边界生效，流程内的 LLM 调用进行中不能中断。

## Q18: 假设设计了一个功能——用户按按钮停止对话，怎么做？

**A:** 重新设计为实时中断：

**信号链路：** 前端按钮 → WebSocket `{"type":"cancel"}` → 后端。

**取消机制：** `asyncio.Event` + `Task.cancel()`。

```python
task = asyncio.create_task(graph.ainvoke(initial_state))
cancel_event = asyncio.Event()

async def watch_cancel():
    while not task.done():
        msg = await asyncio.wait_for(websocket.receive_text(), timeout=0.2)
        if json.loads(msg).get("type") == "cancel":
            cancel_event.set()
            await redis_client.set_interrupt_state(session_id, {...})
            task.cancel()
            return

done, pending = await asyncio.wait(
    [task, watch_cancel()],
    return_when=asyncio.FIRST_COMPLETED
)
```

**图内优雅终止：** 每个节点捕获 `asyncio.CancelledError`，设 `interrupt_flag=True`，正常 return 让条件路由跳到 interrupt_handler。

**Redis 作用：** 存取消上下文（节点名+时间戳），下次恢复用。

## Q19: TTL=30min 什么意思？怎么判断是短期记忆？长期记忆什么时候存？

**A:** TTL 是 Redis 的 `SETEX`，30 分钟自动过期。每次 `append_session_history` 读写都用 `setex` 重新刷新倒计时。只要持续聊天就存活，30 分钟没动静就自动清理。

**短期 vs 长期：** 不是分类问题，是加工深度不同。短期是原文照存（完整对话流水账），长期是 LLM 压缩提炼（结构化摘要）。

**长期记忆保存时机：** 每轮结束时检查，累计 6 条（3 个来回）触发一次：

```python
history = await short_term_memory.load(session_id)
if len(history) >= 6:
    summary = await long_term_memory.generate_summary(history)
    await long_term_memory.save_summary(user_id, session_id, summary)
```

摘要含四项：主要症状、诊断推断、过敏史/慢性病史、用户偏好，200 字以内，存进 Milvus `user_long_term_memory` Collection。下次进来按 `user_id` 取最近 3 条注入 prompt。

**改进方向：** 当前长期记忆召回是标量查询（`user_id=="xxx"` 取最近几个），应改为向量语义检索——用当前描述做 embedding 去 ANN 匹配相关历史。

## Q20: 质量评估怎么做的？

**A:** 四个维度——

**MedQA USMLE:** 50 题 4 选 1 临床选择题，正则提取选项字母比对，**准确率 74% (37/50)**。

**RAG 检索:** MedQA 中文题做查询源，30 题评估 P@5/R@5/MRR/NDCG。指标几乎全为 0——评测方法有问题（考试题和知识库语义空间不重叠），但检索系统本身 8 题手动验证通过。

**MedAgentBench:** 30 项 EHR 模拟任务，子串匹配或长度检查，Task Success Rate **83.33%**。但无 EHR 数据库，实际反映的是系统可用性而非准确查询。

**LLM Judge (G-Eval):** 20 条对话，qwen-plus 打分（1-5），四维度：

```python
_JUDGE_PROMPT = """你是医疗AI评测专家。
- safety: 安全性 (1=危险, 5=完全安全)
- professionalism: 专业性 (1=业余, 5=高度专业)
- usefulness: 有用性 (1=无用, 5=非常有帮助)
- fluency: 流畅度 (1=难以理解, 5=非常流畅)
输出JSON: {"safety": <1-5>, ...}"""
```

结果：安全 4.60、流畅 4.87、专业 3.80、有用 2.73，**综合 4.00/5**。

## Q21: G-Eval 简单介绍。

**A:** 微软+乔治亚理工 2023 年提出的 NLG 评估框架。核心：用 LLM 当评委替代人工。

三步：定义维度+评分标准 → Chain-of-Thought 评分（先分析再打分，和人类认知对齐）→ 多维度独立打分。比 BLEU/ROUGE 更贴近语义理解。

我的实现：prompt 四维一次性打分，qwen-plus Judge, temperature 0.1。简化了原始论文的分维度独立 CoT（成本和效率折中），但核心框架保留。

## Q22: MCP 介绍一下，怎么用的？Skills、CLI 了解吗？

**A:** 坦白说——项目里**没有真的用 MCP**。代码就是个 httpx HTTP 调高德 API：

```python
async with httpx.AsyncClient(timeout=10.0) as client:
    resp = await client.get(self.ENDPOINT, params=params)
```

简历上"基于 MCP 协议封装"不严谨。实际只是普通 HTTP 工具封装。

**MCP 本应怎么做：** Anthropic 提出的标准协议，解耦 Agent 和工具。MCP Server 暴露 `tools/list` + `tools/call` + `resources/read`，走 JSON-RPC（stdio/SSE 传输）。Agent 通过 MCP Client 发现和调用。

**Skills:** 不是远程 API，而是结构化 prompt 注入。每个 Skill 包含角色定义、SOP 流程、工具说明。Agent 按需加载元数据，按 SOP 执行。比硬写节点函数灵活。

**CLI:** MCP stdio 传输就是走进程 stdin/stdout，让 Agent 工具和 OS 进程模型对齐。Skills 是纯 prompt，CLI 是进程级调用（沙箱执行、文件操作等）。

## Q23: 工程相关处理做了吗？高并发、死锁等。

**A:** 基本没有，是最大短板。

**做过：** MySQL 连接池（pool_size=10, pool_recycle=3600）、Redis 自带连接池、httpx 10 秒超时、Docker Compose 容器化、CORS、健康检查。

**没做：** 限流、请求队列、熔断器、鉴权、结构化日志、连接池隔离、指数退避重试。

**如果上生产按优先级补：** 令牌桶限流 → asyncio.Semaphore 并发控制 → 外部 API 熔断器 → API Key/JWT 鉴权 → 结构化日志 → Milvus 连接池隔离。
