### 问题 1：Hatchling 构建配置错误 —— 构建工具无法确定 Wheel 文件内容

**严重度**：🔴 阻塞（项目无法安装）

**现象**：
```
ValueError: Unable to determine which files to ship inside the wheel
```

**根因分析**：
`pyproject.toml` 中使用了 `[tool.uv]` 配置段，但切换到 hatchling 构建后端后，构建工具无法自动发现需打包的源文件。Hatchling 不使用 `[tool.uv]` 配置。

**解决步骤**：

1. **查阅** Hatchling 官方文档确认 wheel 目标配置格式
2. **修改** `pyproject.toml`：
   ```toml
   # 移除
   [tool.uv]
   ...

   # 添加
   [build-system]
   requires = ["hatchling"]
   build-backend = "hatchling.build"

   [tool.hatch.build.targets.wheel]
   packages = ["src"]
   ```
3. **验证**：`pip install -e .` 成功

**影响文件**：[pyproject.toml](file:///e:/sword art online/Documents/medagent/pyproject.toml)

**最佳实践**：
- 切换构建后端时，必须同时更新其专属配置段
- Hatchling 不会自动从 `[tool.uv]` 继承配置
- 明确指定 `packages = ["src"]` 是 src-layout 项目的最佳实践

### 问题 2：Python 3.14 与 pandas/pymilvus 库兼容性冲突

**严重度**：🔴 阻塞（核心库无法导入）

**现象**：
```
ModuleNotFoundError: No module named 'pandas._libs'
```

**根因分析**：
`uv sync` 默认使用系统 Python 3.14，该版本对部分 C 扩展包（pandas, pymilvus）的二进制分发尚未稳定，导致底层 C 扩展模块编译/链接失败。

**排查过程**：
1. 运行 `uv pip list` 确认 pandas 已安装但导入失败 → 初步判断为编译问题
2. 运行 `python --version` → Python 3.14.0a1（预发布版本）
3. 运行 `pip install --force-reinstall pandas` → 仍在 3.14 下失败
4. 结论：Python 3.14 太新，依赖库未适配

**解决方案**：

```powershell
# 方案1：指定 Python 3.12 安装
uv sync --python "C:\Users\sword art online\AppData\Local\Programs\Python\Python312\python.exe"

# 方案2：虚拟环境重建
Remove-Item -Recurse -Force .venv
uv venv --python 3.12
uv sync
```

选择方案2，确保虚拟环境基础解释器版本锁定。

**影响文件**：`.venv` 虚拟环境

**最佳实践**：
- 在 `pyproject.toml` 中声明 `requires-python = ">=3.10,<3.13"` 防止意外使用预发布版本
- C 扩展依赖较多的项目，应锁定稳定 Python 版本号
- 开发前检查 `python --version`，避免使用 alpha/beta 版本

### 问题 4：FastAPI 导入 main 模块导致循环依赖

**严重度**：🔴 阻塞（服务无法启动）

**现象**：
执行 `verify_design.py` 时，第16节 `from src.main import app` 触发：
```
Redis 连接失败 → 异常在模块级别传递 → 整个验证脚本崩溃
```

`main.py` 在模块级别实例化了 `redis_client.connect()` 调用，在 [lifespan 上下文中](file:///e:/sword art online/Documents/medagent/src/main.py) 执行。但任何 `from src.main import app` 都会触发整个模块的顶层代码执行，包括异步连接操作，在同步上下文中导致异常。

**根因分析**：
FastAPI 的 `lifespan` 设计就是解决这个问题的——连接操作应在里面做。但之前的部分代码或验证脚本在 `import` 阶段就触发了连接。

**解决步骤**：
1. **审查** `main.py` 确认连接逻辑已全部在 `lifespan` 异步上下文中
2. **审查** 路由文件确认无模块级别的连接调用
3. **修改** 验证脚本，在导入 FastAPI 相关模块时捕获连接异常
4. **确认** 导入不再阻塞

**影响文件**：[src/main.py](file:///e:/sword art online/Documents/medagent/src/main.py)，验证脚本

**最佳实践**：
- 模块顶层代码必须是纯定义，不得执行 I/O 操作
- FastAPI 的数据库/缓存连接必须在 `lifespan` 内完成
- 验证脚本应使用 `try/except` 包裹会触发 I/O 的导入

### 问题 5：意图分类器输出 JSON 格式不稳定

**严重度**：🟡 影响质量（可能导致运行时崩溃）

**现象**：
原版 `classify_intent` 使用 `json.loads(raw)` 直接解析 LLM 输出，当 LLM 返回：
- ` ```json\n{...}\n``` `（markdown 代码块包裹）
- `分类结果是 {...}`（混入中文描述）
- `{"intent": bad, "confidence": 0.95}`（JSON 内缺少引号）
- 纯文本胡说八道

会导致 `json.JSONDecodeError` 崩溃，且无任何容错。

**根因分析**：
1. System Prompt 约束不够严格——只说了"仅返回JSON"，未禁止 markdown 包裹和额外文本
2. `temperature=0.1` 仍有随机性
3. 解析方式单一——只处理了 ````json` 前缀，未处理其他变体

**解决步骤**：

1. **强化 System Prompt**：4 条严格规则 + 温度降至 0.0 + max_tokens 降至 64
2. **实现三层容错提取** `_extract_json()`：
   - 第一层：正则提取 ````json` / ```` 代码块内容
   - 第二层：直接解析整段文本
   - 第三层：正则匹配 `{...}` JSON 对象
3. **实现字段校验** `_validate_result()`：
   - intent 不在 6 个合法值内 → 回退 unknown
   - confidence 超出 [0, 1] → 钳制到边界
   - 缺少字段 → 默认值填充
4. **5 种场景 + 4 个边界条件** 单元测试全部通过

**影响文件**：[src/intent/classifier.py](file:///e:/sword art online/Documents/medagent/src/intent/classifier.py)

**最佳实践**：
- 永远不要信任 LLM 的输出格式，必须有多层防御
- 正则提取 > json.loads > 字段校验 三层保险
- temperature=0.0 对分类任务是最佳选择
- max_tokens 应设为恰好够用的值，防止 LLM 自由发挥

### 问题 6：端口 8000 被占用

**严重度**：🟢 操作问题

**现象**：
```
ERROR: [WinError 10013] 以一种访问权限不允许的方式做了一个访问套接字的尝试。
```

**根因分析**：
上一个 uvicorn 进程未完全退出，端口仍处于 `TIME_WAIT` 或 `LISTEN` 状态。

**解决方案**：

```powershell
# 一键释放
$p = (Get-NetTCPConnection -LocalPort 8000 -ErrorAction SilentlyContinue).OwningProcess
if ($p) { Stop-Process -Id $p -Force }
```

**影响文件**：无

**最佳实践**：
- 启动前先检查端口占用，或使用 `--reload` 时确保旧进程已停止
- Windows 下 `netstat -ano | findstr :8000` + `taskkill /PID xxx /F` 等效
- 可写入启动脚本自动检测

### 问题 7：Docker Hub 在中国大陆无法直连

**严重度**：🟡 环境问题

**现象**：
```
Error response from daemon: failed to resolve reference "docker.io/minio/minio:latest":
dial tcp 157.240.8.50:443: connectex: A connection attempt failed
```

**根因分析**：
`registry-1.docker.io` 在中国大陆被 GFW 阻断，无法直接 DNS 解析和 TCP 连接。

**解决步骤**：

1. 打开 Docker Desktop → Settings → Docker Engine
2. 配置镜像加速器：
   ```json
   {
     "registry-mirrors": [
       "https://docker.m.daocloud.io",
       "https://dockerhub.timeweb.cloud",
       "https://docker.1panel.dev"
     ]
   }
   ```
3. Apply & Restart Docker Desktop
4. 重新执行 `docker compose up -d`

**影响文件**：Docker Desktop 配置（非项目文件）

**最佳实践**：
- 项目文档中应注明中国大陆用户需配置镜像加速
- 备选方案1：阿里云容器镜像服务（`cr.console.aliyun.com` → 镜像加速器）获取专属地址
- 备选方案2：手动预先 `docker pull` 各镜像后再 `compose up`

### 问题 9：意图分类器实时 API 调用失败

**严重度**：🟡 影响功能（多智能体协作关键节点）

**现象**：
服务启动后发送测试消息，响应包含：`"请稍后重试"` 或无意义输出。

**根因分析**：
意图分类器 (`classify_intent`) 的 LLM API 调用异常被 `except Exception` 静默吞掉，回退 intent 为 `unknown`。包括但不限于：
- API Key 权限不足
- API 限流 / QPS 超限
- 网络超时
- 响应格式非法

**解决步骤**：

1. **添加详细日志**：判断具体是哪种异常
   ```python
   except Exception as e:
       logger.error(f"Intent classification failed: {e}")
       state["intent"] = "unknown"
       state["intent_confidence"] = 0.0
       state["timeout_flag"] = True
   ```
2. **降低 temperature 到 0.0**：减少输出随机性
3. **降低 max_tokens 到 64**：加快响应，减少超时
4. **入参 JSON 化**：`json.dumps({"role": "user", "message": user_message})` 结构化输入

**影响文件**：[src/intent/classifier.py](file:///e:/sword art online/Documents/medagent/src/intent/classifier.py)，[src/agents/graph.py](file:///e:/sword art online/Documents/medagent/src/agents/graph.py)

**最佳实践**：
- 异步 LLM 调用需要全面的异常捕获和降级策略
- 错误日志比默默回退更有价值——优先记录，再降级
- JSON 结构化输入有助于 LLM 理解和响应一致性

### 问题 10：PowerShell 变量名被转义破坏

**严重度**：🟢 工具链

**现象**：
```powershell
$env:TRAE_SANDBOX_COMMAND_LINE = '.venv\Scripts\uvicorn.exe ... 2>&1'
```
PowerShell 在解析命令时对等号、引号、管道符等特殊字符处理与 bash 不同，导致字符串被截断或参数异常。

**根因分析**：
PowerShell 的字符串转义规则与 POSIX shell 不同：
- `2>&1` 在 PowerShell 中不是标准重定向语法
- `$env:` 变量赋值中引号嵌套容易混淆

**解决步骤**：
使用 PowerShell 原生语法替代：
```powershell
# 代替 bash 的 2>&1
*>&1

# 或使用 Start-Process 带 -RedirectStandardError
```

**影响文件**：无（仅命令行操作）

**最佳实践**：
- PowerShell 环境下避免使用 POSIX shell 语法
- 长命令优先写入脚本文件再执行
- 使用 `--%` 停止解析标志避免转义问题

### 问题 11：路由文件闭包引用问题

**严重度**：🟡 架构问题

**现象**：
路由文件 [routes/chat.py](file:///e:/sword art online/Documents/medagent/src/routes/chat.py) 和 [routes/ws.py](file:///e:/sword art online/Documents/medagent/src/routes/ws.py) 在函数内通过 `from src.main import ...` 延迟导入，这种模式在模块初始化时可能导致引用不明确。

**根因分析**：
起初路由函数内部通过局部导入引用 `graph`（如在 `chat_endpoint` 内写 `from src.agents.graph import graph`），这在 FastAPI 的依赖注入和测试场景下容易引发闭包引用不一致。

**解决步骤**：
将 graph 等依赖提升到模块级别导入：
```python
# 不佳实践（在函数内导入）
async def chat_endpoint(...):
    from src.agents.graph import graph
    result = await graph.ainvoke(...)

# 良好实践（模块级别）
from src.agents.graph import graph

async def chat_endpoint(...):
    result = await graph.ainvoke(...)
```

**影响文件**：[src/routes/chat.py](file:///e:/sword art online/Documents/medagent/src/routes/chat.py)，[src/routes/ws.py](file:///e:/sword art online/Documents/medagent/src/routes/ws.py)

**最佳实践**：
- 所有导入放在文件顶部（模块级别）
- 避免在函数/类方法内部做 `import`
- 如果确实需要延迟导入（如避免循环引用），用 `TYPE_CHECKING` + 类型注解

### 问题 12：Schemas 模块循环引用

**严重度**：🟡 架构问题

**现象**：

**解决步骤**：
```python

from src.schemas.models import (
    ChatRequest, ChatResponse, WSMessage, WSAck,
    SessionHistory, SessionResponse,
    HealthResponse, JSONResponse,
    INTERNAL_ERROR_RESPONSE, HEALTH_OK_RESPONSE,
)
```

**影响文件**：[src/schemas/__init__.py](file:///e:/sword art online/Documents/medagent/src/schemas/__init__.py)

**最佳实践**：
- `__init__.py` 中使用显式导入（`from x import A, B`）代替 `import *`
- 数据模型类集中在一个文件，避免跨文件引用
- 常量/响应预设放在单独的位置

### 问题 14：Python 安装路径包含空格导致 uv 路径解析异常

**严重度**：🟢 环境限制

**现象**：
```
uv sync --python "C:\Users\sword art online\AppData\Local\Programs\Python\Python312\python.exe"
```
路径中 `sword art online` 包含空格，某些 shell 解析时可能截断路径或产生歧义。

**根因分析**：
Windows 用户目录名包含空格是常见情况（如 `C:\Users\Firstname Lastname`），uv 和 pip 通常能正确处理引号包裹的路径，但在某些交叉调用场景下（如构建脚本在子进程中执行），引号可能被消耗，导致空格被解释为参数分隔符。

**解决策略**：
1. 始终使用双引号包裹路径
2. 备选方案：创建无空格的符号链接
   ```powershell
   New-Item -ItemType Junction -Path "C:\Python312" -Target "C:\Users\sword art online\AppData\Local\Programs\Python\Python312"
   ```
3. 或使用 Python Launcher：`py -3.12` 代替完整路径
